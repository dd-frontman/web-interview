# HTTP/3

**HTTP/3** — это HTTP поверх **QUIC** (а QUIC поверх UDP), а не поверх TCP.

Главная цель: убрать сетевые задержки, которые в HTTP/2 оставались из-за TCP.

## Что даёт HTTP/3

### 1. Меньше задержек при потере пакетов

В TCP потеря пакета может затормозить всё соединение.

В QUIC потоки обрабатываются отдельно, поэтому потеря данных в одном потоке не блокирует остальные так жёстко, как в TCP-сценарии.

### 2. Быстрее установка соединения

В QUIC криптография встроена (TLS 1.3), поэтому handshake обычно быстрее, чем цепочка TCP + TLS в старых схемах.

### 3. Connection migration

QUIC лучше переносит смену сети (например, Wi‑Fi -> LTE), что особенно полезно на мобильных устройствах.

## Базовая схема работы

1. Браузер подключается к серверу по UDP (QUIC).
2. Договаривается о шифровании (TLS 1.3 внутри QUIC).
3. Запускает HTTP/3-потоки.
4. Параллельно может поддерживаться fallback на HTTP/2, если инфраструктура не поддерживает `h3`.

## Практика внедрения

### На сервере/edge

- включить поддержку QUIC/HTTP/3;
- оставить HTTP/2 как fallback;
- отдать `Alt-Svc`, чтобы браузер знал про `h3`.

Пример (Nginx c поддержкой quic):

```nginx
server {
  listen 443 ssl http2;
  listen 443 quic reuseport;

  ssl_protocols TLSv1.3;
  add_header Alt-Svc 'h3=":443"; ma=86400';
}
```

### Проверка

- DevTools -> Network -> `Protocol`.
- Ищем `h3` у запросов.

## Ограничения и подводные камни

- не все корпоративные сети хорошо относятся к UDP-трафику;
- часть инфраструктуры (старые балансировщики, WAF, прокси) может не поддерживать QUIC;
- нужна аккуратная эксплуатация и мониторинг fallback на `h2`.

## Когда выбирать HTTP/3

- публичные сайты с мобильным трафиком;
- high-latency сети и регионы с нестабильным интернетом;
- когда уже есть зрелая edge-инфраструктура (CDN/proxy) с поддержкой `h3`.
