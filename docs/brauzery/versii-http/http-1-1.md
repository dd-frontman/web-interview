# HTTP/1.1

**HTTP/1.1** стандартизирован в 1999 году и очень долго был основой веба.  
Это текстовый протокол поверх TCP: клиент отправляет запрос, сервер возвращает ответ.

## Что дал HTTP/1.1 по сравнению с ранними версиями

- `keep-alive` по умолчанию (не нужно открывать новый TCP-коннект на каждый файл)
- обязательный заголовок `Host`
- `chunked transfer encoding` для потоковой отдачи
- лучшее кэширование через `Cache-Control`, `ETag`, `Last-Modified`

## Как выглядит обмен

```http
GET /products/42 HTTP/1.1
Host: example.com
Accept: application/json
Connection: keep-alive
```

```http
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 27

{"id":42,"name":"Keyboard"}
```

Главная особенность: всё читаемо человеком, но из-за текстового формата и модели обмена есть накладные расходы.

## Где начинаются проблемы производительности

### 1. Ограниченный параллелизм

Браузер обычно держит ограниченное число TCP-соединений на домен (часто около 6).  
Если ресурсов много, часть запросов ждёт в очереди.

### 2. Head-of-line на уровне HTTP/1.1

На одном соединении запросы/ответы идут последовательно.  
Если ранний ответ задержался, последующие тоже ждут.

### 3. Повторение заголовков

В каждом запросе заново передаются похожие заголовки (`Cookie`, `User-Agent`, `Accept`), что увеличивает трафик.

## Когда HTTP/1.1 всё ещё нормальный выбор

- внутренние сервисы в простой инфраструктуре
- старые прокси/балансировщики без поддержки HTTP/2/3
- сценарии, где у клиента и так мало параллельных запросов

## Мини-чеклист, если остаёшься на HTTP/1.1

- включить `keep-alive`
- использовать `gzip`/`brotli`
- настроить кэширование (`Cache-Control`, `ETag`)
- уменьшать количество запросов к одному origin
