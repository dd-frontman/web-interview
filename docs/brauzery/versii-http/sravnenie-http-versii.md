# Сравнение HTTP/1.1 HTTP/2

Это страница-итог: когда выбирать HTTP/1.1, когда HTTP/2, и когда есть смысл переходить на HTTP/3.

## Кратко

- **HTTP/1.1**: старый и совместимый, но менее эффективный при большом числе запросов.
- **HTTP/2**: текущий стабильный дефолт для большинства проектов.
- **HTTP/3**: лучший для нестабильных сетей и мобильного трафика, если инфраструктура готова.

## Таблица сравнения

| Критерий | HTTP/1.1 | HTTP/2 | HTTP/3 |
| --- | --- | --- | --- |
| Транспорт | TCP | TCP | QUIC (UDP) |
| Формат | Текстовый | Бинарный | Бинарный |
| Параллельные запросы | Ограничены соединениями | Мультиплексирование в одном соединении | Мультиплексирование в QUIC |
| Сжатие заголовков | Нет стандартизованного механизма уровня протокола | HPACK | QPACK |
| Head-of-line blocking | Выраженный на уровне HTTP | На уровне TCP всё ещё возможен | Сильно снижен за счёт QUIC |
| Handshake | TCP + TLS | TCP + TLS | QUIC + TLS 1.3 |
| Работа в мобильных сетях | Хуже | Лучше 1.1 | Обычно лучше всех |
| Совместимость инфраструктуры | Максимальная | Очень высокая | Зависит от поддержки QUIC |

## Практический выбор

### Когда брать HTTP/2

- почти всегда как базовый стандарт;
- есть HTTPS и обычный веб-трафик;
- нужна хорошая производительность без сложной миграции.

### Когда идти в HTTP/3

- много мобильного трафика;
- нестабильные сети и высокий RTT;
- CDN/edge уже поддерживают `h3`.

### Когда остаётся HTTP/1.1

- legacy-среда;
- старые прокси/балансировщики;
- ограничения инфраструктуры, где `h2/h3` пока недоступны.

## План миграции без боли

1. Включить и стабилизировать HTTP/2.
2. Добавить HTTP/3 как дополнительный режим (не вместо `h2`, а вместе).
3. Мониторить в проде долю `h3`, latency, ошибки и fallback.
4. Подкрутить CDN/proxy конфиги по результатам телеметрии.

## Где проверить версию

- Chrome/Edge/Firefox DevTools -> Network -> `Protocol`
- значения обычно: `http/1.1`, `h2`, `h3`
