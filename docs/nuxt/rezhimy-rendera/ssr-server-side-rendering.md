**SSR (Server-Side Rendering)** — это способ рендеринга, когда **HTML-страница формируется на сервере**, а не в браузере.  
Браузер получает уже готовый HTML, а затем поверх него «гидратируется» JavaScript (оживляет UI, добавляет интерактивность).

---

## 2) Как работает шаг за шагом

1. **Клиент делает запрос** к серверу: `GET /products/123`.
2. **Сервер**:
   - выполняет код приложения (Vue/React на сервере через Node/SSR-движок).
   - получает данные из API/БД.
   - собирает HTML со всеми данными.
3. **Ответ**: сервер отдаёт готовый HTML (видимый текст/контент).
4. **Браузер**:
   - сразу отображает страницу (даже без JS).
   - загружает JS-бандл.
   - выполняет **hydration** → «привязывает» обработчики событий к уже готовой разметке.

---

## 3) Отличие SSR от CSR

- **CSR (Client-Side Rendering):**
  - Сервер отдаёт «пустой» `index.html` с `<div id="app"></div>`.
  - Данные и UI загружаются через JS.
  - Первая отрисовка медленнее (нужно дождаться JS).
  - SEO — хуже (поисковики видят пустую страницу, если не умеют ждать).

- **SSR:**
  - Контент формируется на сервере.
  - Страница видна сразу (быстрее TTFB и LCP).
  - SEO-friendly (поисковики видят готовый HTML).
  - Минус: больше нагрузка на сервер, сложнее кеширование.

---

## 4) SSR в Vue (Nuxt)

Пример на **Nuxt 3**:

```vue
<script setup lang="ts">
const { data: product } = await useAsyncData("product", () => $fetch(`/api/products/123`));
// этот код выполнится на сервере, HTML с продуктом придёт сразу
</script>

<template>
    <div>
        <h1>{{ product.name }}</h1>
        <p>{{ product.price }} ₽</p>
    </div>
</template>
```

---

## 5) SSR в React (Next.js)

```tsx
export async function getServerSideProps() {
    const res = await fetch("https://api.example.com/products/123");
    const product = await res.json();
    return { props: { product } };
}

export default function Product({ product }: any) {
    return (
        <div>
            <h1>{product.name}</h1>
            <p>{product.price} ₽</p>
        </div>
    );
}
```

## 6) SSR vs SSG vs ISR

| Подход  | Когда генерируется                          | Где хранится                             | Для чего подходит                      |
| ------- | ------------------------------------------- | ---------------------------------------- | -------------------------------------- |
| **CSR** | В браузере                                  | Нужен только `index.html` + JS           | SPA, дашборды, закрытые кабинеты       |
| **SSR** | При каждом запросе                          | Сервер формирует заново                  | SEO-сайты с часто обновляемыми данными |
| **SSG** | На этапе билда                              | Готовые статические HTML-файлы           | Блоги, документация, маркетинг         |
| **ISR** | При первом запросе + регенерация по таймеру | CDN/сервер хранит сгенерированную версию | E-commerce, каталоги, контентные сайты |

---

## 7) Плюсы SSR

- Быстрое отображение первого контента (**лучше TTFB/LCP**).
- SEO-оптимизация (готовый HTML для поисковиков).
- Легче шарить ссылки (в соцсетях сразу видно заголовок/описание).
- Универсальный код (одна логика на сервере и клиенте).

---

## 8) Минусы SSR

- Нагрузка на сервер (HTML строится на каждый запрос).
- Сложнее кешировать (динамический контент).
- Код сложнее (hydrate, состояние, гонки данных).
- Дольше «time-to-interactive», т.к. гидрация занимает время.

---

## 9) Когда стоит применять SSR

✅ Да:

- Маркетинговые страницы, блоги, СМИ (важно SEO).
- Интернет-магазины, каталоги (быстрый первый рендер + SEO).
- Когда нужно, чтобы контент был доступен без JS (например, для скринридеров, простых браузеров).

❌ Нет:

- Внутренние кабинеты, админки (SEO не нужен).
- Одностраничные приложения, где всё крутится на API и пользователь всегда авторизован.

# Шпаргалка

| Характеристика           | CSR             | SSR                                |
| ------------------------ | --------------- | ---------------------------------- |
| Рендеринг                | В браузере      | На сервере                         |
| HTML при загрузке        | Пустой div + JS | Готовый контент                    |
| Скорость первого рендера | Медленнее       | Быстрее                            |
| SEO                      | Плохой          | Отличный                           |
| Нагрузка на сервер       | Минимальная     | Высокая                            |
| Кеширование              | Легко (CDN)     | Сложнее                            |
| Где использовать         | Дашборды, SPA   | Публичные сайты, e-commerce, блоги |
