## Способы авторизации / аутентификации — обзор

| Метод                                                                 | Что это / как работает                                                                                                                                                                                                                                         | Когда удобно использовать                                                                                                                                                                                | Плюсы                                                                                                                                                                                   | Минусы                                                                                                                                                                                                                      |
| --------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Сессии + куки (Session-/Cookie-based auth)**                        | Пользователь логинится, сервер создаёт сессию, устанавливает cookie. При каждом запросе браузера куки автоматически отправляются, сервер проверяет сессию.                                                                                                     | Для классических веб-приложений, где важно держать состояние пользователей на сервере; страницы, которые рендерятся на сервере.                                                                          | Простой подход; куки управляются браузером; позволяет быстро аннулировать сессию; легче защищать от некоторых атак, если настроить HTTP-only, Secure, SameSite.                         | Требует хранения сессий на сервере; нагрузка при большом числе пользователей; сложнее масштабировать; CSRF-атаки; проблемы с распределёнными серверами без общей сессии / хранилища.                                        |
| **JWT / Token-based авторизация**                                     | После логина сервер отдаёт клиенту токен (часто JWT). Клиент хранит токен (в localStorage, memory, или в куке) и отправляет его в заголовке `Authorization: Bearer <token>` при запросах.                                                                      | Одностраничные приложения (SPA), мобильные приложения, когда нужно API, распределённые сервисы, где сервер не хочет держать состояние сессий.                                                            | Stateless (сервер не держит состояние сессии); легко масштабировать; можно использовать один токен в разных сервисах; часто быстрее проверка токена по подписи без чтения сессии из БД. | Управление жизненным циклом токенов (истечение, refresh); безопасность хранения (XSS, если хранить в localStorage); сложность аннулирования токена (revocation); возможная нагрузка, если JWT большие или часто передаются. |
| **OAuth 2.0 + OpenID Connect (OIDC)**                                 | Протоколы для авторизации / аутентификации через внешний провайдер либо identity server. OAuth используется для делегированной авторизации, а OpenID Connect добавляет слой идентификации (входа, профиля пользователя). | Когда нужно логин через Google / Facebook / другой провайдер; когда требуется Single Sign-On; когда у приложения есть внешние API и нужно управление доступом; когда хочется не хранить все пароли себя. | Позволяет использовать сторонний провайдер; хорошо зарекомендовано в индустрии; стандартизировано; позволяет разделять ответственность (аутентификация / авторизация).                  | Настройка сложнее; redirect URI / безопасность токенов; необходимо конфигурировать scopes, узнать identity provider; возможны ошибки безопасности, если неверно реализовать; дополнительная инфраструктура.                 |
| **Многофакторная аутентификация (MFA / 2FA)**                         | После базовой аутентификации дополнительно требуется второй фактор: SMS/OTP, приложение-генератор кодов, аппаратный ключ, биометрия и др.                                                                                                                      | Когда требуется повышенная безопасность (банковские приложения, административные панели, приложения с чувствительными данными)                                                                           | Существенно повышает защиту; даже если пароль скомпрометирован, нужен второй фактор; многие пользователи привыкли к MFA.                                                                | Усложнён UX; возможны сбои / неудобства (второй фактор недоступен, телефон разряжен и др.); требует дополнительной инфраструктуры; возможны задержки/стоимости при SMS и др.                                                |
| **Без пароля (Password-less, Magic Link, WebAuthn / Passkeys / OTP)** | Вместо пароля: отправка ссылки по email, временный код, WebAuthn (аппаратный ключ / биометрия). Пользователь подтверждает личность, используя link / устройство.                                                                                               | Когда хочется убрать пароли, упростить UX; когда приложения могут требовать высокий уровень безопасности; когда пользователи предпочитают “войти по ссылке” или по устройству.                           | Удобство; уменьшение рисков, связанных с управлением паролями; если используется WebAuthn / Passkeys — высокий уровень безопасности; меньше уязвимостей от утечек паролей.              | Требует поддержки браузерами / устройствами; возможно сложнее реализовать “восстановление” доступа; ссылки или коды могут быть фишинг-мишенью; более сложное управление, если пользователь теряет устройство / email.       |

## Применение: как выбрать и сочетать

- Если приложение SPA / API, вероятно JWT + Refresh токены или использование OAuth/OIDC будет хорошим вариантом.
- Если сайт с серверным рендером и высокой критичностью безопасности, сессии + cookie могут быть проще.
- Часто используют гибридные подходы: например, JWT для API, куки для сайта; или OAuth/OIDC + JWT.
- Обязательно защищать токены / куки: HTTPS, HttpOnly, Secure, SameSite, защита от XSS/CSRF.
- Реализовать механизм “отзыва” токена (revocation) и / или blacklist, когда токен или refresh токен недействителен.

## Дополнитель детали из источников

- OpenID Connect — это слой идентификации поверх OAuth 2.0, добавляющий подтверждение личности + получение профиля пользователя.
- JWT vs сессии: блог Stytch, GeeksforGeeks и другие сравнивают trade-offs между хранением состояния на сервере (сессии) и на клиенте (JWT).
- Важно: JWT не всегда автоматически “лучше” — многое зависит от требований к безопасности и архитектуре.

## Чит-шит

| Метод                    | Stateless / Stateful               | Подходит для                                           | Главное про безопасность                                                      |
| ------------------------ | ---------------------------------- | ------------------------------------------------------ | ----------------------------------------------------------------------------- |
| Сессии + куки            | Stateful (сервер хранит состояние) | Традиционные веб-приложения, сайты с server-render     | Защита cookie, CSRF, очистка сессий, управление сроком жизни                  |
| JWT / токены             | Stateless                          | API, мобильные, microservices, фронтенд/SPA            | Подписи, истечение срока, защита хранилища, refresh / revocation              |
| OAuth2 / OIDC            | сторонний сервер + токены          | SSO, логин через Google/Facebook/IDP                   | Защита redirect URI, scopes, идентификация пользователя, безопасная настройка |
| MFA / 2FA                | добавочный уровень безопасности    | приложения с чувствительными данными / аккаунтами      | Есть резервные пути, безопасные методы второй аутентификации                  |
| Password-less / WebAuthn | часто комбинация методов           | UX-ориентированные приложения, повышенная безопасность | Безопасное хранение ключей / удостоверений, защита link/code, fallback        |
