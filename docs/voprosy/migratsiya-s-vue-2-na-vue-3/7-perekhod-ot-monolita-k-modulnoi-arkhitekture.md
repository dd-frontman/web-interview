# Как вы понимаете переход от монолитной frontend-архитектуры к модульной?

## Ответ

Для меня переход от монолитной архитектуры к модульной — это не просто разложить файлы по разным папкам. Это значит выделить в системе независимые смысловые блоки с понятными границами ответственности.

В монолите обычно наблюдаются такие признаки:

- сильная связность между частями системы;
- общие утилиты используются везде без границ;
- компоненты знают слишком много друг о друге;
- бизнес-логика размазана по приложению;
- сложно понять, где заканчивается одна предметная область и начинается другая.

Модульная архитектура предполагает, что система делится по доменным или feature-границам. У каждого модуля должны быть:

- понятная ответственность;
- ограниченный публичный интерфейс;
- минимальные внешние зависимости;
- изолированная внутренняя логика.

Практически это означает:

- модули `users`, `documents`, `approvals`, `notifications` и так далее;
- локализация бизнес-логики внутри своей предметной области;
- меньше общего "магического" кода;
- меньше неожиданных side-effects между частями системы.

Плюсы такого подхода:

- проще сопровождать;
- проще онбордить новых людей;
- проще тестировать;
- проще менять одну часть системы без каскадных последствий в других.

То есть модульность — это в первую очередь про управляемость изменений, а не про структуру папок саму по себе.
