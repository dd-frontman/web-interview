# Какие архитектурные улучшения имеет смысл делать параллельно с миграцией на новую версию фреймворка?

## Ответ

Миграция — это хорошее окно возможностей для архитектурных улучшений, потому что в этот момент всё равно происходит пересмотр части системы.

Но важно не пытаться улучшить вообще всё сразу. Обычно имеет смысл менять только то, что:

- уже затрагивается миграцией;
- реально мешает развитию;
- снижает предсказуемость системы.

Самые полезные улучшения в этот момент обычно такие.

Во-первых, декомпозиция по доменам. Если проект был устроен как монолит с высокой связностью, миграция — хороший момент, чтобы разложить его на модули по предметным областям.

Во-вторых, отделение UI от бизнес-логики. Часто в старых проектах компоненты перегружены. Во время миграции удобно вынести:

- бизнес-логику в composables/services;
- API-взаимодействие в отдельный слой;
- модели и типы в отдельные модули.

В-третьих, пересборка state management. Если store устроен хаотично, миграция — хороший повод сделать его чище и понятнее.

В-четвёртых, усиление контрактов:

- типизация;
- единые API-модели;
- унификация паттернов.

Главное правило здесь — архитектурные улучшения должны снижать стоимость будущих изменений, а не просто делать код "красивее".
