# Правила написания документации

Этот проект состоит в первую очередь из контента, поэтому для новых материалов важны не только смысл и точность, но и единый формат. Часть правил ниже проверяется автоматически скриптами, часть является редакционным стандартом, который стоит соблюдать для новых страниц.

## Что считается нормой для нового материала

- Новый файл создаётся в `docs/<section>/<slug>.md`.
- Путь и имя файла пишутся в `kebab-case`.
- Для всех страниц, кроме `docs/index.md`, обязателен единый `frontmatter`.
- Новый материал должен быть либо доступен из sidebar, либо иметь входящие ссылки из других страниц. Иначе страница считается orphan.
- Контент пишется обычным markdown. Автогенерируемые файлы вручную не редактируются.

## Где размещать новые страницы

- Обычная статья: `docs/<section>/<slug>.md`
- Вложенная тема: `docs/<section>/<group>/<slug>.md`
- Страница-обзор раздела: `docs/<section>/index.md` или корневой файл раздела, если такой паттерн уже используется в разделе
- Главная страница `docs/index.md` является исключением: это VitePress-страница с HTML/Vue-разметкой, для неё стандартный `frontmatter` не обязателен

Примеры:

- `docs/react/index.md`
- `docs/vue/defineexpose.md`
- `docs/voprosy/praktika-vue-i-reaktivnost/1-kogda-ispolzovat-computed-watch-watcheffect.md`

## Обязательный frontmatter

Для всех обычных страниц используйте такой каркас:

```yaml
---
title: "Название статьи"
description: "Короткое описание страницы в 1-2 предложениях."
tags:
  - "section-tag"
  - "topic-tag"
updatedAt: "2026-03-14"
---
```

Правила по полям:

- `title`: человекочитаемый заголовок страницы. Для новых материалов лучше, чтобы совпадал с `# H1`.
- `description`: короткое содержательное описание. Лучше 1-2 предложения без ссылок, HTML и служебного текста.
- `tags`: массив строк. Обычно достаточно 2-5 тегов. Теги влияют на поиск и автоподбор связанных тем.
- `updatedAt`: дата последнего содержательного обновления в формате `YYYY-MM-DD`.
- `search: false`: опционально, только для очень больших шпаргалок или страниц, которые не нужно включать в локальный поиск.

Если `frontmatter` неполный, `yarn sync:frontmatter` попытается его нормализовать автоматически, но для новых страниц лучше заполнять его сразу руками.

## Предпочтительный паттерн статьи

В репозитории есть старые и новые страницы разного качества. Для новых материалов стоит ориентироваться на текущий устойчивый паттерн:

1. `frontmatter`
2. `# H1`, совпадающий с `title`
3. Короткий вводный абзац: что это, зачем нужно, в каком контексте тема важна
4. Основные смысловые блоки с `##`
5. Списки и примеры кода там, где это помогает понять тему быстрее
6. При необходимости блок с официальными ссылками
7. В конце страницы блок связанных материалов

Шаблон обычной статьи:

```md
---
title: "Название статьи"
description: "Короткое описание страницы."
tags:
  - "section"
  - "topic"
updatedAt: "2026-03-14"
---
# Название статьи

Короткий вводный абзац на 2-4 строки.

## Что это

Короткое объяснение сути темы.

## Когда применять

- случай 1;
- случай 2;
- случай 3.

## Пример

```ts
const example = true;
```

<OfficialDocsLinks
    :links="[
        { title: 'Официальная документация', href: 'https://example.com' },
    ]"
/>

---

<RelatedTopics
    :items="[
        { title: 'Связанная тема 1', href: '/section/topic-1' },
        { title: 'Связанная тема 2', href: '/section/topic-2' },
        { title: 'Связанная тема 3', href: '/section/topic-3' },
    ]"
/>
```

## Паттерн для страниц-вопросов

Для материалов в `docs/voprosy/**` хорошо читается такой формат:

- `title` и `# H1` формулируются как полноценный вопрос
- после заголовка идёт `## Ответ`
- ответ начинается с прямого тезиса, а не с длинного вступления
- дальше идут короткие аргументы, списки, примеры и практические оговорки

Шаблон:

```md
---
title: "Когда использовать X?"
description: "Короткий ответ о том, когда использовать X и чем он отличается от альтернатив."
tags:
  - "voprosy"
  - "section"
  - "topic"
updatedAt: "2026-03-14"
---
# Когда использовать X?

## Ответ

Короткий прямой ответ на вопрос.

### Когда подходит

- пункт 1;
- пункт 2.

### Когда не подходит

- пункт 1;
- пункт 2.

---

<RelatedTopics
    :items="[
        { title: 'Связанная тема', href: '/section/related-topic' },
    ]"
/>
```

## Паттерн для обзорных страниц раздела

Обзорная страница раздела обычно:

- кратко объясняет, что входит в раздел;
- даёт порядок чтения;
- выделяет базовые статьи для собеседований;
- заканчивается `RelatedTopics`.

Для новых обзорных страниц предпочтителен явный `# H1`, даже если в старых файлах встречаются варианты без него.

## Стиль текста

- Пишите по-русски, короткими абзацами, без воды.
- Начинайте с сути, а не с общего рассуждения.
- Объясняйте термин, затем практический смысл, затем ограничения и trade-offs.
- Там, где это полезно, приводите код, таблицу, список типичных ошибок или критерии выбора.
- Используйте одинаковые термины по всей странице.
- Не оставляйте следы диалога с ИИ: фразы вроде `если хочешь, могу`, `скажи, что нужно дальше`, дисклеймеры про ChatGPT и похожий служебный текст недопустимы.

## Ссылки и навигация

- Для внутренних ссылок предпочтительны абсолютные роуты вида `/vue/defineexpose`.
- Не делайте self-link на текущую страницу без причины.
- Не добавляйте битые ссылки, дубли в `RelatedTopics` и ссылки на несуществующие роуты.
- Если страница не попадает в sidebar автоматически как часть структуры, обеспечьте на неё входящие ссылки из других материалов.

## Встроенные блоки

В markdown можно использовать зарегистрированные Vue-компоненты:

### `RelatedTopics`

Используйте внизу страницы, когда хотите явно задать curated-список связанных материалов. Если ручного блока нет, тема всё равно может получить автоподбор связанных страниц по тегам.

```vue
<RelatedTopics
    :items="[
        { title: 'Связанная тема', href: '/section/topic' },
    ]"
/>
```

### `OfficialDocsLinks`

Используйте, когда есть одна или несколько действительно полезных официальных ссылок.

```vue
<OfficialDocsLinks
    :links="[
        { title: 'MDN: Object.freeze()', href: 'https://developer.mozilla.org/...' },
    ]"
/>
```

## Что точно ломает валидацию

- отсутствие обязательного `frontmatter`;
- пустой или слишком плохой `title`;
- пустой `tags` или `updatedAt` не в формате `YYYY-MM-DD`;
- незакрытый fenced code block;
- Obsidian-ссылки `[[...]]` и `![[...]]`;
- артефакты вида `oaicite` и `utm_source=chatgpt.com`;
- битые внутренние ссылки;
- orphan-страницы без sidebar и входящих ссылок.

## Чек-лист перед коммитом

1. Убедиться, что файл лежит в правильном разделе и имеет понятный slug.
2. Проверить `frontmatter`, `title`, `description`, `tags`, `updatedAt`.
3. Проверить внутренние ссылки и блок `RelatedTopics`.
4. Обновить `updatedAt`, если материал менялся по смыслу.
5. Запустить проверки.

Минимальный набор команд:

```bash
yarn sync:meta
yarn validate-content
yarn validate-links
yarn validate-routes
yarn lint:md
```

Дополнительно перед крупным обновлением полезно запускать:

```bash
yarn report:stale
yarn build
```
